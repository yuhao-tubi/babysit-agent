Thread
Lifecycle A Thread is the unit of triage. It carries one Verdict, one gate run, and one attempt-loop guard — and it moves through six states. Here is the machine, and exactly where you can put your hands on it.

Poll cycle→ verdict · Agent SDK decides→ gate · objective self-check→ executor · push · reply · escalate

01 The machine

States & transitions

Solid cyan lines are autonomous — the daemon walks them on its own each poll cycle. Dashed amber lines only move when you act from the dashboard. Two terminal states loop all the way back to pending: the lifecycle is idempotent and recomputed from live GitHub state, never from first sight.

02 Where you put your hands on it

Operator controls, by state

Every Thread shows Mark resolved and Reply on GitHub. The state-specific moves are below. Approve is the sole push path — nothing else writes code to a PR branch (except author classes opted into autoPushClasses).

awaiting_approval

A frozen Proposal is parked. It carries up to two independently-approvable parts — the change (gate-passed code diff or PR-description rewrite) and the reply. Each is approved on its own; the Thread resolves only once both are settled.

  • Approve & push code The sole push path — change part only. Re-checks the frozen diff against current HEAD, re-runs the gate, then fast-forward pushes — or for a description Proposal, runs gh pr edit. Posts no reply.→ resolved if no reply pending, else stays awaiting_approval
  • Post reply Posts the drafted reply verbatim to the GitHub thread — independent of the change. Post it without the code, or push the code without it.→ resolved if change settled, else stays awaiting_approval
  • Dismiss Drop the drafted reply without posting (you'll reply by hand, or none is needed). Settles the reply part.→ resolved if change settled, else stays awaiting_approval
  • Send to agent A freeform Instruction always re-proposes a fresh diff to review. It never pushes.→ in_progress → awaiting_approval
  • Reply on GitHub Post text verbatim to the thread, no agent.→ resolved
blocked

A decision is needed (escalation, loop-guard trip, or a too-large change parked as a Manual plan). It fired a notification and waits for you.

  • Send to agent Your Instruction re-proposes. Escalation Options are one-click chips that pre-fill this box — selecting one doesn't run the agent by itself.→ in_progress → awaiting_approval
  • Copy plan For a Manual plan: copy the self-contained brief and run it in Claude Code on the branch yourself. The daemon never pushes it.→ stays blocked
  • Reply / Mark resolved Answer verbatim, or close it out by hand.→ resolved
error

A transient failure was recorded — the daemon stays up. Recover from the most durable artifact.

  • Retry Resume from the last state. If a frozen Proposal exists it re-applies it (apply-check + re-gate); otherwise it falls back to a fresh rerun. Never replays a stored Verdict decision.→ in_progress
  • Fresh Rerun Reset and re-run the whole Verdict pipeline from scratch.→ pending
resolved

Done. But not frozen — new feedback can legitimately re-open it.

  • Fresh Rerun Manually reopen and re-decide.→ pending
  • ↻ auto-reopen New non-self activity reopens it next poll. Activity by @me — including the agent's own acks — never counts.→ pending
pendingin_progress

Transient, daemon-owned states. The pipeline drives them — you rarely intervene, but you still can.

  • Mark resolved Abandon triage at any point; resolves the GitHub thread if inline.→ resolved
  • Reply on GitHub Step in with a direct human reply.→ resolved

03 Control reference

Every button, one row each

Each dashboard control, the endpoint it calls, and exactly what it does to the Thread. A → resolved / awaiting_approval result means it settles one part of a two-part Proposal: resolved once the other part is also done, else it stays parked for the remaining part. All GitHub writes honor dryRun (default true).

ControlEndpointWhat it doesResult
Approve & push code POST /approve The sole push path — change part only. Re-applies the frozen change: apply-check + re-gate + ff-push for code; gh pr edit for a description. Posts no reply. → resolved / awaiting_approval
Post reply POST /approve-reply Posts the Proposal's drafted reply part verbatim to the GitHub thread — independent of the change. No agent, no push. → resolved / awaiting_approval
Dismiss POST /dismiss-reply Settles the Proposal's reply part without posting it (owner replies by hand, or none needed). No push. → resolved / awaiting_approval
Send to agent POST /instruct Freeform Instruction always re-proposes. Prefix reply: to park that text as a reply Proposal to review & Post (never posted directly); ignore to drop. Never pushes. → awaiting_approval
Reply on GitHub POST /reply Posts the text verbatim to the GitHub thread (no agent, no verdict), then closes the Thread. → resolved
Mark resolved POST /resolve Marks the Thread resolved; resolves the underlying GitHub thread too if it's an inline comment. → resolved
Retry POST /retry Recover after a transient failure from the durable artifact: re-apply a frozen Proposal, else fall back to a fresh rerun. → in_progress
Fresh Rerun POST /rerun Reset to pending and re-run the entire Verdict pipeline from scratch. → pending
GUARDRAIL

Approve is load-bearing. A code change is always a Proposal — built, gate-verified, parked — and pushes only on your Approve, re-checked against current HEAD first. A Proposal's reply is a separate part: Approving the change pushes but posts no reply, and Posting the reply doesn't push — the Thread resolves only once both are settled. risk:"high" keeps the Proposal but vetoes any auto-push. amend_pr_body and Manual plans are never auto-applied. A reply: Instruction parks its text as a reply Proposal (verbatim, no agent) to review and Post — it never posts directly. Restart recovery re-renders durable artifacts; it never replays a stored Verdict decision.
